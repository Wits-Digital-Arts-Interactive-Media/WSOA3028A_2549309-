Interaction Design and User Experience

1 March 2024 by Aaminah Habib

Exploring the history of the internet through readings and examining the first-ever published website was interesting. It's remarkable to witness the transformation from a simple text-based interface to the vibrant, multimedia-rich experiences we encounter today. The evolution of user interfaces, with the addition of colors, images, links, and interactive elements like buttons, has undoubtedly enhanced the user experience and accessibility of the web. Vannevar Bush's visionary paper continues to resonate, advocating for human-centric design principles that mirror the associative trails of the human mind, ultimately shaping the way we navigate and comprehend information online (Bush, 1945).

In the digital world, every pixel and code snippet play a crucial role in storytelling. The narrative isn't just in the content we produce but, in the journey, we craft for our users. This week, I dove into the art of interaction design to align my website's content, experience, and interface elements with the intricate expectations of diverse audiences. This includes potential employers and clients who seek a testament to skill and creativity, fellow professionals and collaborators or aspiring professionals and students seeking knowledge and inspiration. My portfolio is more than a showcase; it's a convergence point for dialogue, learning, and professional growth.

Related to URL scheme, Semantic Markup of my structure

    The exercise emphasizes understanding the audience to craft an intuitive, informative, and engaging experience. This aligns with the principle of semantic markup, which aims to create web content that is meaningful and accessible to both users and search engines. By structuring your content around the needs and expectations of the users, I can ensure that the website is not only visually appealing but also functionally relevant.

  • Information Structure and Interface Elements

    The detailed breakdown of my website's sections (Home Page, Portfolio, Blog, Essays, About Page) reflects a thoughtful approach to information architecture. This structure should be mirrored in the URL scheme, with clear, descriptive paths that correspond to each section (e.g., /portfolio, /blog, /about). Such a scheme enhances usability and SEO, as it makes the website's organization logical and easy to navigate for both users and search engines. Using semantic HTML tags for further reinforces the structure and meaning of the content.

    Semantic markup practice improves accessibility, allowing screen readers and assistive technologies to better interpret the content. In the context of this exercise, applying semantic markup to my site's structure (e.g., marking up the portfolio section with tags, individual works with tags, and navigation links with tags) makes the website more accessible and understandable to all users.

JavaScript Organization

During the development of my website, I faced the challenge of managing multiple JavaScript files within my js folder. I pondered whether to keep them separate or to group related functions into fewer files. To make an informed decision, I conducted some research to identify the best practices that would suit my needs.

Keeping Functions in Separate Files

Pros:
  • Organization: Each file has a single responsibility, making it easier to understand and maintain.
  • Modularity: Specific functionalities can be managed independently without affecting others.
  • Scalability: Adding new features or functions is straightforward, avoiding clutter in existing files.
  • Debugging: Isolating issues in a single file is simpler.
Cons:
  • Complexity: Managing multiple files can be cumbersome for smaller projects.
  • Performance: Loading many small files might affect performance, though this can be mitigated with bundling tools like Webpack.

Regrouping Functions

Pros:
  • Simplicity: Fewer files to manage can make the project seem less complex.
  • Readability: Related functions in one file can provide a comprehensive view of related functionalities.
  • Performance: Fewer files can improve load times.
Cons:
  • Maintainability: Larger files can become harder to navigate and maintain.
  • Conflict: Higher chances of merge conflicts in version control systems.

Given the size and scope of my project, I concluded that combining related functions into fewer files would simplify the structure while ensuring that each file still has a clear purpose. This approach balances the benefits of both strategies, providing an organized yet manageable codebase.

Here's the plan:

  • navigation.js: For all navigation-related functionalities (menus, breadcrumbs).
  • form.js: For form handling (Formspree, modals).
  • utilities.js: For utility functions (scrolling, back-to-top button).

By implementing this structure, I aim to maintain clarity and simplicity in my project, making it easier to manage, debug, and expand in the future.
